# AudioFlow Phase 2 开发规划

## 目标：打通真实实时数据主链路，建立“可运行、可稳定、可优化”的核心系统

---

## 一、Phase 2 的定位

如果说 Phase 1 是：

* 搭工程底座
* 拆 UI 结构
* 建边界
* 立规矩

那么 Phase 2 的目标就是：

> **把“实时音频数据 → 处理 → 渲染 → 控制回写”这条主链路真正打通，并且做成后续可持续优化的系统。**

这阶段不是简单地“接上 WebSocket 就完了”，而是要解决四个核心问题：

1. **真实 WebSocket 数据如何稳定接入**
2. **高频数据如何在前端内安全流转**
3. **Canvas 如何在不依赖 React 重渲染的前提下持续绘制**
4. **设置项如何双向联动，并具备可观测性与可恢复性**

---

## 二、Phase 2 的核心目标

### 必须达成

* 真实 WebSocket 链路接入
* 消息解析与运行时校验建立
* 高频音频数据进入独立 runtime，不污染 React 主渲染链路
* Canvas 使用真实数据驱动绘制
* 设置项可变更并回写服务端
* 连接状态、异常状态、重连状态可展示
* 建立基础性能基线和排障信息

### 阶段结束后的可演示结果

* 页面打开后自动建立连接
* 可实时显示频谱
* 参数面板变更可影响服务端或本地处理逻辑
* 断线后可自动重连，并在 UI 上可见
* 页面持续运行 20~30 分钟后没有明显内存泄漏或性能恶化
* 主流桌面环境下体验稳定

### 明确非目标

Phase 2 仍然**不追求**：

* 多可视化模式
* WebGL
* 完整测试覆盖率
* 高级主题系统
* PWA
* 复杂动画/视觉精修
* 大规模组件抽象平台化

---

## 三、资深前端视角下的 Phase 2 架构原则

---

### 原则 1：实时主链路必须与 React 渲染彻底解耦

这条原则在 Phase 2 必须落地，不再只是设计共识。

#### 错误做法

```text
WebSocket → Zustand → React render → Canvas update
```

这条链路一旦成立，后面性能、抖动、调试、扩展都会出问题。

#### 正确做法

```text
WebSocket → protocol decode → audio runtime → Canvas RAF render
                              ↓
                       lightweight UI state
```

也就是：

* React 只订阅低频状态
* Canvas 直接读取 runtime 最新快照
* 高频帧不进入 React 组件 diff 主链路

---

### 原则 2：数据链路必须具备“防脏、防抖、防堆积”能力

真实服务接入后，前端面对的不是“理想数据”，而是：

* 消息乱序
* 非法字段
* 帧率波动
* 网络抖动
* 服务端偶发空帧
* 突发高频消息

所以 Phase 2 必须引入：

* 运行时协议校验
* 丢帧容忍
* 只保留最新帧
* 渲染与接收解耦
* 错误兜底

不能让前端渲染链路被上游数据质量绑架。

---

### 原则 3：先追求“稳定正确”，再追求“理论最优”

大厂工程里，实时页面第二阶段最常见误区就是过早优化：

* 一上来就 OffscreenCanvas
* 一上来就 Worker
* 一上来就 WebGL
* 一上来就复杂 ring buffer 调度

如果当前数据量只是普通频谱条形图，Phase 2 的正确目标是：

* 链路正确
* 生命周期清晰
* 内存稳定
* 性能可测
* 后续可扩展

不是炫技。

---

### 原则 4：所有关键链路必须可观测

Phase 2 开始，不能再靠肉眼看“好像正常”。

必须能回答这些问题：

* 当前连接状态是什么
* 收到消息频率是多少
* 当前渲染 FPS 是多少
* 最近一帧时间戳是多少
* 是否发生过消息解析失败
* 是否发生过重连
* 设置项是否已成功下发

这决定后续线上定位问题的效率。

## 四、Phase 2 总体架构设计

---

## 1. 目标数据流

```text
WebSocket Server
    │
    ▼
websocketService
    │
    ├── connection status events
    ├── raw message events
    ▼
protocol decoder / validator
    │
    ├── invalid -> error reporter
    ▼
audioProcessor
    │
    ├── normalize
    ├── db / peak / avg
    ├── smoothing / weighting
    ▼
audioRuntime
    │
    ├── latest magnitudes
    ├── frame version
    ├── timestamps
    ├── stats snapshot
    ▼
SpectrumRenderer (RAF loop)
    │
    ▼
Canvas 2D

settingsStore
    │
    ▼
settingsSyncController
    │
    ├── send config to websocket
    └── local runtime apply
```

---

## 2. 模块边界设计

### A. `websocketService`

职责：

* connect / disconnect
* reconnect
* heartbeat
* status emit
* raw message emit
* send config

不做：

* 不做 React state 更新
* 不做业务解析
* 不做 UI 提示

---

### B. `protocol decoder`

职责：

* 把原始 message 转成前端协议对象
* 做 schema 校验
* 做版本兼容
* 做非法消息拒绝

建议：

* 用 `zod` 或轻量 schema
* 至少校验：

    * message type
    * magnitudes 是否存在
    * 数组长度是否合法
    * timestamp 是否有效

---

### C. `audioProcessor`

职责：

* 纯计算
* 数据归一化
* 派生指标计算
* smoothing / attack / decay / weighting

要求：

* 尽量纯函数
* 尽量不持有全局状态
* 可单测

---

### D. `audioRuntime`

职责：

* 持有高频运行时数据
* 提供 `updateFrame`
* 提供 `getSnapshot`
* 提供 `getStats`
* 提供轻量订阅能力

要求：

* 不依赖 React
* 不直接触发 UI render
* 可选择复用 typed array，控制 GC

---

### E. `renderer`

职责：

* 维护 RAF 生命周期
* 从 runtime 拉最新数据
* 做绘制
* 计算 FPS
* 处理 resize / cleanup

要求：

* 只依赖 canvas context 和 runtime
* 可被 React 外壳挂载/卸载
* 无业务耦合

---

### F. `settingsSyncController`

职责：

* 监听 settingsStore 变化
* 做本地 apply
* 向 websocket 下发配置
* 处理发送失败 / ack / 回滚策略

这是 Phase 2 很容易漏掉的层。
不建议让组件自己直接 `onChange -> websocket.send()`，那会把 UI 和协议再次耦死。

---

## 五、Phase 2 关键技术设计

---

## 1. WebSocket 连接设计

### 推荐状态机

```ts
type ConnectionStatus =
  | 'idle'
  | 'connecting'
  | 'connected'
  | 'reconnecting'
  | 'disconnected'
  | 'error';
```

### 必备能力

* 首次连接
* 主动断开
* 被动断开
* 自动重连
* 最大重连次数或指数退避
* 心跳超时检测
* 状态事件广播

### 推荐策略

* 重连延迟：指数退避 + 上限
* 心跳检测：30s 左右量级，具体按服务端协议
* 进入后台页签时不强制断开，但需验证浏览器行为
* 页面 unload 时显式 cleanup

### 验收标准

* 网络抖动后能恢复
* 重连期间 UI 有明确状态
* 不会出现重复连接和多实例并存

---

## 2. 协议与消息建模

### 建议消息拆分

不要只靠一个“万能 JSON”。

推荐至少定义：

* `spectrum_frame`
* `connection_ack`
* `settings_ack`
* `error`
* `heartbeat`

### 示例类型

```ts
type IncomingMessage =
  | { type: 'spectrum_frame'; timestamp: number; magnitudes: number[]; peakDb?: number; avgDb?: number }
  | { type: 'settings_ack'; requestId: string; success: boolean }
  | { type: 'heartbeat'; timestamp: number }
  | { type: 'error'; code: string; message: string };
```

### 关键建议

* 协议字段命名在 Phase 2 定死
* 所有可选字段明确语义
* 错误码不要只给 message，最好有 code
* 给设置下发保留 `requestId`

这会大幅降低后续联调成本。

---

## 3. 高频数据存储策略

这里是 Phase 2 的性能核心。

### 不建议

每收到一帧都：

* `new Float32Array(...)`
* `setState`
* 让 React 感知 frame 变化

### 更建议

`audioRuntime` 内部持有可复用缓冲区：

* `magnitudesBuffer`
* `frameVersion`
* `lastFrameTimestamp`
* `peakDb`
* `avgDb`

更新时：

* 尽量复用数组内存
* 写入后 version++
* renderer 每帧读取最新 version

### 关键原则

前端只需要“最新帧”，不是“每一帧历史都必须进 UI”。

也就是：

* **接收频率** 与 **绘制频率** 分离
* 数据积压时优先丢旧帧，保留最新帧

这是实时可视化页面最典型的工程决策。

---

## 4. 渲染器设计

### 推荐结构

* `SpectrumCanvas.tsx`：React 壳
* `createSpectrumRenderer.ts`：真正渲染器
* `audioRuntime`：数据读取源

### 渲染器职责

* 初始化 context
* 监听 DPR
* resize 适配
* 清屏
* 绘制频谱条
* 绘制 peak 线/辅助线
* 采集 FPS
* cleanup

### 关键约束

* 不依赖 React state
* 不在 RAF 里创建大量对象
* 不在每帧重复计算静态坐标
* 文本绘制频率控制，避免不必要开销

### 推荐的优化顺序

1. 正确性
2. 生命周期
3. 对象分配优化
4. 静态几何预计算
5. 降低重复绘制成本
6. 再考虑 offscreen/worker

---

## 5. 设置项同步设计

这部分要避免“表面能用，后面不可维护”。

### 推荐链路

```text
UI Control
  ↓
settingsStore.update()
  ↓
settingsSyncController
  ├── local apply
  └── websocket send config
```

### 为什么要单独一层

因为设置项未来会越来越复杂：

* 有的只影响本地渲染
* 有的要发送给服务端
* 有的要等待 ack
* 有的失败后需要回滚
* 有的要节流发送

如果一开始就直接在组件里 send，后面一定散。

### 推荐策略

* slider 类设置做轻量节流
* select / mode 切换可即时发送
* 失败时保留错误提示入口
* 重要配置带 requestId

### 输出物

* `audioRuntime.ts`

## 九、Phase 2 的核心验收指标

建议在这一阶段就明确几个工程指标，而不是只说“看起来挺流畅”。

### 建议指标

* 端到端感知延迟：目标维持在业务可接受范围内，建议以 `< 50ms` 为设计目标
* FPS：主流桌面环境下接近稳定 60，至少不应频繁跌出可接受阈值
* 重连恢复：断网/服务重启后应能自动恢复
* 长时间运行：20~30 分钟内内存不持续线性上涨
* 错误可见：协议错误、连接错误、发送失败可被记录和展示

这里不要求一上来全部满分，但必须开始量化。

## 十、Phase 2 主要风险与缓解

### 风险 1：真实协议和规划假设不一致

**表现**：字段缺失、命名变化、帧格式不稳定。
**缓解**：decoder + schema 校验 + 协议适配层，不让业务代码直接吃原始消息。

---

### 风险 2：高频数组更新导致 GC 抖动

**表现**：偶发卡顿、长时间运行后帧率波动。
**缓解**：尽量复用 typed array，减少每帧 new object。

---

### 风险 3：React 无意中重新卷入高频链路

**表现**：某个组件订阅了 frame 级数据，整体性能骤降。
**缓解**：明确 code review 红线，高频数据不得直接进普通 UI store selector。

---

### 风险 4：设置项发送过于频繁

**表现**：slider 拖动时 send 风暴。
**缓解**：引入节流、合并发送、区分本地预览与最终提交。

---

### 风险 5：断线重连逻辑导致多实例并存

**表现**：重复消息、重复渲染、状态错乱。
**缓解**：service 内部单实例控制，connect/disconnect 状态机收敛。

---

## 十一、Phase 2 的技术决策清单

这部分是 Tech Lead 应该在进入开发前明确拍板的。

### 决策 1

**高频音频帧进入 runtime，不进入 React store 主路径。**

### 决策 2

**协议消息必须做运行时校验。**

### 决策 3

**设置项发送走 controller，不允许组件直接调用 websocket.send。**

### 决策 4

**渲染器只能依赖 runtime 和 canvas，不允许反向依赖组件状态。**

### 决策 5

**出现帧堆积时，优先丢旧帧，保最新帧。**

### 决策 6

**优先保证稳定和可观测，再进入高阶性能优化。**
